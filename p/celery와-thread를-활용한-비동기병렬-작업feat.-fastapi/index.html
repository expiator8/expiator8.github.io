목표 시나리오

FastAPI로 진행
이미지를 7장을 생성하고 생성 결과를 s3에 업로드하는 api를 작성
이미지를 생성하기 위해서는 외부 이미지 서버에 요청을 해야함.
이미지 서버에 이미지 생성 최대 병렬 요청 가능한 클라이언트 수는 5,
s3에 업로드 최대 병렬 요청 가능한 수는 25로 설정.
이 작업들을 비동기로 수행해 때문에 실제로 모두 완료되기까지 시간이 오래 걸리지만
바로 완료가 된 것처럼 메시지를 Response 해야함.

다시말해 api 서버는 여러가지 10000개의 api 요청을 병렬로 받을 수도 있지만 이미지를 생성하는 특정 api들에 대해서는 최대 5개까지의 요청을 처리하고 s3 업로드는 최대 25개까지의 처리가 되어야한다!

import time
from concurrent.futures import ThreadPoolExecutor
from fastapi import FastAPI


image_server_executor = ThreadPoolExecutor(max_workers=5)
s3_upload_executor = ThreadPoolExecutor(max_workers=25)


app = FastAPI(debug=True)


def generate_images():
    """Generate images from Image server"""
    print("start generate image")
    # 이미지 서버에 이미지 생성 요청. 10초 걸린다고 가정
    time.sleep(10)
    return [
        "Image1",
        "Image2",
        "Image3",
        "Image4",
        "Image5",
        "Image6",
        "Image7",
    ]


def upload_image(image):
    """Upload Image to S3"""
    # S3에 업로드 요청. 5초 걸린다고 가정
    print("start image upload")
    time.sleep(5)
    return f"upload complete {image}"


@app.get("/slow")
def slow():
    print("start slow api")
    # 이미지 생성 작업: 최대 5개의 작업을 병렬로 실행
    images = image_server_executor.submit(generate_images).result()

    # s3 업로드 작업: 이미지 생성 작업의 결과(7개의 이미지)를 사용해 최대 25개의 작업을 병렬로 실행
    uploaded_images = [
        s3_upload_executor.submit(upload_image, image) for image in images
    ]

    return {"message": "slow task response"}

시간이 걸리는 이유는 submit().result()이다.
generate_images의 결과가 완료될 때까지 기다렸다가 upload_image 작업을 할 수 있기 때문에
response를 받기까지 generate_images의 수행 시간 10초 가량이 걸린다.

import asyncio
import aiohttp
import ssl


async def fetch(session, url, ssl_context):
    try:
        async with session.get(url, ssl=ssl_context) as response:
            if response.headers["Content-Type"] == "application/json":
                return await response.json()  # JSON 응답을 반환받는 상황 가정
            else:
                return await response.text()  # JSON이 아닌 응답 처리
    except Exception as e:
        return {"error": str(e)}


# 모든 API 호출을 실행하고 결과를 수집
async def fetch_all(urls):
    ssl_context = ssl.create_default_context()
    ssl_context.check_hostname = False
    ssl_context.verify_mode = ssl.CERT_NONE
    async with aiohttp.ClientSession() as session:
        tasks = [fetch(session, url, ssl_context) for url in urls]
        # 결과를 완료되는대로 즉시 출력
        for task in asyncio.as_completed(tasks):
            result = await task
            print(result)
        # 결과를 모두 완료된후 한번에 출력할 경우
        # results = await asyncio.gather(*tasks)
        # return results


# 비동기 작업을 시작하는 메인 함수
async def main():
    urls = ["http://localhost:5000/slow"] * 5 + ["http://localhost:5000/simple"] * 5
    results = await fetch_all(urls)


# 비동기 메인 함수 실행
asyncio.run(main())

반면 예를들어 gunicorn에서 동일한 simple, slow api에 각각 5개의 병렬 요청을 보내면 gunicorn에서는 다음과 같이 동작한다.

요청을 동기 처리하는 gunicorn에서는 slow api를 처리하는 동안 다른 api 호출이 지연된다.
즉, slow api를 제외한 모든 api가 아무리 빠르게 처리가 가능하더라도 slow api에 영향을 받아 지연될 수 있다.

*이정도 요청만으로도 nginx에서 timeout을 조정해주기 전엔 504 Gateway Time-out 에러가 발생했다.

gunicorn 실행시 worker와 thread 등의 설정으로 어느정도 지연 해소는 가능하다.

1

gunicorn --workers 1 --threads 5

예를들어 이 경우에는 slow api가 5개의 thread를 전부 사용한다면 그다음 api 호출부터 지연된다.

uvicorn이 gunicorn의 worker 프로세스로서 동작하게 할수도 있는데 Kubernetes나 Docker Swarm로 프로세스 관리자 역할을 한다면 uvicorn 단일 프로세스로만 사용하면 좋다고 하는 것 같다.

1

gunicorn main:app --worker-class uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000

이런식으로 하면 fastapi에서 gunicorn을 사용하면서도 비동기로 처리된다. 상황에 맞게 잘 사용하면 될것같다.

그런데 uvicorn.workers.UvicornWorker를 사용할때 –threads 옵션을 설정하면 요청을 받았을때 threads 개수만큼 먼저 비동기로 시작하고 다음 작업을 처리할까 싶었는데 기대한대로 동작하진 않았다.

7개의 요청을 보냈는데 5개를 처리하고 다음 2개를 처리하는게 아니라 들어온 요청을 모두 처리한다.

import time
import logging
from concurrent.futures import ThreadPoolExecutor
from celery import Celery
from fastapi import FastAPI


logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)


celery_app = Celery(
    "test",
    broker="redis://127.0.0.1:6379/0",
    backend="redis://127.0.0.1:6379/0",
)
celery_app.conf.update(broker_connection_retry_on_startup=True)


def generate_images():
    logger.info("Task started")
    time.sleep(10)
    logger.info("Task completed")
    return [
        "Image1",
        "Image2",
        "Image3",
        "Image4",
        "Image5",
        "Image6",
        "Image7",
    ]


def upload_image(image):
    """Upload Image to S3"""
    # S3에 업로드 요청. 5초 걸린다고 가정
    logger.info("start image upload")
    time.sleep(5)
    return f"upload complete {image}"


@celery_app.task
def generate_image_task():
    # 이미지 생성 작업: 최대 5개의 작업을 병렬로 실행
    future = image_server_executor.submit(generate_images)
    images = future.result()
    # s3 업로드 작업: 이미지 생성 작업의 결과(7개의 이미지)를 사용해 최대 25개의 작업을 병렬로 실행
    uploaded_images = [
        s3_upload_executor.submit(upload_image, image) for image in images
    ]


image_server_executor = ThreadPoolExecutor(max_workers=5)
s3_upload_executor = ThreadPoolExecutor(max_workers=25)


app = FastAPI(debug=True)


@app.get("/slow")
def slow():
    print("start slow api")
    generate_image_task.delay()
    return {"message": "slow task response"}


@app.get("/simple")
def simple():
    print("start simple api")
    time.sleep(1)
    return {"message": "Simple api response"}

우선 slow api로 7번의 요청을 보냈을때 모든 요청이 빠르게 response 되고 simple, slow api 모두 각각 7번의 요청을 보냈을때도 response가 지연되지 않는다
